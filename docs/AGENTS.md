# ADSys documentation agent guide

This guide applies to work under `docs/`. Follow the repository-level
instructions in `../AGENTS.md` as well as the documentation-specific guidance
here.

## What this documentation is

ADSys documentation is a Sphinx project that uses MyST Markdown,
`canonical-sphinx`, and the extensions listed in `requirements.txt`. Read the
Docs builds the website using `.readthedocs.yaml`.

A selected set of pages is also embedded in the ADSys binary through `docs.go`.
The daemon serves this content as the offline documentation available through
`adsysctl doc`.

The content follows the [Diátaxis](https://diataxis.fr/) framework:

- `tutorial/` contains learning-oriented, end-to-end lessons
- `how-to/` contains task-oriented procedures
- `explanation/` contains concepts, architecture, and background
- `reference/` contains factual descriptions of commands, features, policies,
  and terminology

Supporting content lives in:

- `images/` for screenshots and other static images
- `diagrams/` for Mermaid source files
- `reuse/` for shared reStructuredText links and substitutions
- `pro_content_notice.txt` for the notice included on Ubuntu Pro pages
- `.sphinx/` for templates, static assets, and documentation tooling

Key files include:

- `index.md` for the landing page and top-level navigation
- `how-to/index.md`, `explanation/index.md`, and `reference/index.md` for
  section navigation; the tutorial is linked directly through
  `tutorial/getting-started.md`
- `conf.py` for Sphinx extensions, redirects, and build hooks
- `Makefile` and `requirements.txt` for local builds and checks
- `.custom_wordlist.txt` for accepted technical terms
- `docs.go` for the offline-documentation embed patterns
- `make_toctree.sh` for generated policy navigation

The embed patterns currently include `index.md`, direct Markdown children of
the four Diátaxis directories, and `how-to/certificates/*.md`. They do not
include generated policy pages. Update `docs.go` when adding another nested
directory that must be available through `adsysctl doc`.

## Generated content

Do not directly edit generated reference pages:

- `reference/adsysctl-cli.md`, `reference/adsysd-cli.md`, and
  `reference/adwatchd-cli.md` are generated from Cobra command definitions by
  `go generate ./cmd/adsysd`
- `reference/policies/` is generated by `admxgen` and the policy-build workflow
  from policy definitions and keys collected from supported Ubuntu releases
- `make_toctree.sh` rewrites the indexes below
  `reference/policies/Computer Policies/` and
  `reference/policies/User Policies/` when Sphinx initializes

Change the relevant command definition, `admxgen` definition, or generator
instead of generated Markdown. Regenerate only when generated artifacts are
part of the requested change; otherwise follow `CONTRIBUTING.md` and leave
post-merge refreshes to project automation. Do not keep policy-index changes
that result only from running a documentation build.

## Build and check

Run documentation commands from the repository root:

```text
make -C docs install
make -C docs run
make -C docs html
make -C docs lint-md
make -C docs vale
make -C docs spelling
make -C docs linkcheck
make -C docs pa11y
```

`install` creates `docs/.sphinx/venv` and requires the system
`python3-venv` package. Use `run` for an interactive preview at
`http://127.0.0.1:8000`.

Use `TARGET` to restrict Vale-based checks while iterating:

```text
make -C docs vale TARGET=how-to/set-up-adsys.md
make -C docs spelling TARGET=how-to/set-up-adsys.md
```

Run `html` after navigation, configuration, MyST syntax, or cross-reference
changes. Run `linkcheck` when changing links or moving pages. Run `pa11y` for
changes to templates, layout, or other accessibility-sensitive content.

## Writing conventions

### Language

- Use US English, active voice, and second person where practical
- Expand uncommon acronyms on first use
- Use descriptive link text instead of phrases such as "click here"
- Avoid filler such as "simply", "just", "easy", "obviously", and "basically"
- Do not use emojis

Do not make cosmetic rewrites or synonym substitutions unless the task
identifies a concrete accuracy, consistency, or clarity problem.

### Page structure

- Use kebab-case filenames for new hand-written pages
- Add `myst.html_meta.description` front matter to new user-facing pages
- Use one level-one heading per page and sentence case for headings
- Do not skip heading levels, and introduce a section before adding subsections
- Add a unique MyST target before the level-one heading when a page will be
  cross-referenced; follow neighboring labels, which commonly use the
  `tutorial::`, `howto::`, `exp::`, and `ref::` namespaces
- Keep tutorials, how-to guides, explanations, and references focused on their
  respective Diátaxis purpose

Use the same opening structure as current user-facing pages:

```text
---
myst:
  html_meta:
    description: "Describe the page for search results."
---

(howto::page-label)=
# Page title
```

When adding a page, include it in the nearest `index.md` toctree. Update the
top-level `index.md` only when the page belongs in the landing-page navigation.
Hand-written section indexes use `:titlesonly:` toctrees, while the top-level
toctree is hidden. Keep toctree entries in the existing `page`,
`section/index`, or `Title <target>` forms because
`internal/adsysservice/doc.go` parses them to build the `adsysctl doc` chapter
list.

### MyST and Markdown

- Use existing MyST roles such as `{term}`, `{ref}`, `{guilabel}`, `{kbd}`, and
  `{menuselection}` when they fit
- Use `{term}` only for terms defined in `reference/glossary.md`
- Prefer `{ref}` or `[descriptive text](ref::target)` for an existing labeled
  internal target
- In content pages, use relative Markdown links when no labeled target exists;
  preserve the site-root links used by the landing page in `index.md`
- Do not hard-code the published ADSys documentation URL for an internal link
- Use triple-backtick MyST directive fences, matching the syntax already used
  in this project
- Use `{terminal}` for command sessions that show input and output; use `text`
  or another appropriate language fence for input-only commands, code,
  configuration, and logs
- Put inline commands, file paths, option names, and literal values in
  backticks
- Use admonitions sparingly
- Include `pro_content_notice.txt` with the existing `{include}` pattern on
  pages that document Ubuntu Pro features
- Give every image meaningful alternative text
- Keep image paths under `images/` aligned with the content section when
  practical
- Store Mermaid source in `diagrams/`, use the shared theme documented in
  `diagrams/readme.md`, and reference it with the `{mermaid}` directive

Add valid technical terms to `.custom_wordlist.txt` as plain entries instead
of hiding spelling errors with inline markup. Keep case variants together and
preserve the file's existing organization.

## Moving or deleting pages

Preserve inbound links whenever a published page moves or is removed. Add an
entry to the `redirects` mapping in `conf.py`, update affected toctrees and
cross-references, and check the resulting links. If the page is part of offline
documentation, update `docs.go` and verify that the `adsysctl doc` navigation
still resolves it.

## Scope discipline

Keep documentation changes tied to the requested behavior or identified
problem. Prefer official upstream documentation and man pages as sources, and
do not reproduce large sections of external content.
